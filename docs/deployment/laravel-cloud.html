✅ ข้อดีสำหรับโปรเจคนี้

  • Cron + Queue ทำงาน reliable 100% (booking reminders ตรงเวลาแน่นอน)
  • LINE webhook ไม่ timeout ตอน traffic peak
  • Cloudflare R2 + AWS Rekognition — ใช้ต่อได้ไม่ต้องเปลี่ยน
  • Deploy ใหม่ใช้เวลา 2 นาที vs 30 นาทีบน Hostinger
  • ไม่ต้องเจอปัญหา document root อีก (Cloud รู้จัก Laravel)
  • Logs + monitoring built-in ไม่ต้องติด NewRelic/Sentry แยก

⚠️ ข้อจำกัด

  • แพงกว่า Hostinger 5-10x (~฿1,400 vs ~฿200/เดือน)
  • MariaDB → MySQL/Postgres (ต้อง migrate data)
  • Filesystem ephemeral — ต้องใช้ R2/S3 เก็บไฟล์ทั้งหมด (มีอยู่แล้ว ✓)
  • เปลี่ยน workflow — ใช้ git-based deploy แทน FTP/cPanel
  • Vendor-specific dashboard (แต่ code ยังพอร์ตได้)

✅ สิ่งที่ต้องเตรียม

1 Push code ขึ้น GitHub

⏱ ~3 นาที

Laravel Cloud ใช้ GitHub-based deploy เท่านั้น — ดังนั้น code ต้องอยู่บน GitHub repo ก่อน

1.1 สร้าง repo ใหม่บน GitHub

  1. เข้า github.com/new
  2. ชื่อ repo: jabphap (หรือชื่ออะไรก็ได้)
  3. เลือก Private (สำคัญ — มี secrets ใน history)
  4. ห้ามเลือก "Add README" — เราจะ push existing project

1.2 Push code จากเครื่อง local

PS> cd C:\xampp\htdocs\photo-gallery-tailwind
PS> git init
PS> git branch -M main
PS> git add .
PS> git commit -m "initial commit"
PS> git remote add origin git@github.com:YOUR_USERNAME/jabphap.git
PS> git push -u origin main
⚠️ สำคัญ: ตรวจสอบให้แน่ใจว่า .env อยู่ใน .gitignore — ห้าม commit credentials! 
PS> type .gitignore | findstr ".env"
# ควรเห็น: .env

2 แก้ config สำหรับ Cloud

⏱ ~5 นาที

2.1 เพิ่ม cloud.yaml ที่ root

Laravel Cloud อ่านไฟล์นี้เพื่อเข้าใจ build steps ของโปรเจค

name: jabphap

environments:
  production:
    php: "8.3"
    node: "20"

    build:
      - composer install --no-dev --optimize-autoloader
      - npm ci
      - npm run build
      - php artisan config:cache
      - php artisan route:cache
      - php artisan view:cache
      - php artisan event:cache

    deploy:
      - php artisan migrate --force
      - php artisan storage:link
      - php artisan cache:clear

    scheduler:
      enabled: true

    workers:
      default:
        command: php artisan queue:work --queue=default,line,emails --sleep=2 --tries=3 --max-time=3600
        replicas: 1

2.2 เช็ค config/database.php

ต้องรองรับ MySQL 8 — Laravel default รองรับอยู่แล้ว แค่เช็ค charset:

'mysql' => [
    'driver'          => 'mysql',
    'host'            => env('DB_HOST', '127.0.0.1'),
    'port'            => env('DB_PORT', '3306'),
    'database'        => env('DB_DATABASE'),
    'username'        => env('DB_USERNAME'),
    'password'        => env('DB_PASSWORD'),
    'charset'         => 'utf8mb4',        // ✓ รองรับภาษาไทย + emoji
    'collation'       => 'utf8mb4_unicode_ci', // ✓ ตรงกับ Hostinger
    'prefix'          => '',
    'strict'          => true,
    'engine'          => null,
],

2.3 เช็ค config/filesystems.php — R2

โปรเจคนี้ใช้ R2 อยู่แล้ว — แค่ตรวจว่า disk default ไม่ใช่ local

'default' => env('FILESYSTEM_DISK', 'r2'),  // ← ตรงนี้ ไม่ใช่ local

'disks' => [
    'r2' => [
        'driver'   => 's3',
        'key'      => env('R2_ACCESS_KEY_ID'),
        'secret'   => env('R2_SECRET_ACCESS_KEY'),
        'region'   => 'auto',
        'bucket'   => env('R2_BUCKET'),
        'endpoint' => env('R2_ENDPOINT'),
        'use_path_style_endpoint' => true,
    ],
],

2.4 ใช้ Redis สำหรับ session + cache + queue

Cloud มี Redis built-in — ตอน Hostinger เราใช้ file/database driver แต่บน Cloud ใช้ Redis เร็วกว่ามาก

# ใน config/session.php
'driver' => env('SESSION_DRIVER', 'redis'),

# ใน config/cache.php
'default' => env('CACHE_STORE', 'redis'),

# ใน config/queue.php
'default' => env('QUEUE_CONNECTION', 'redis'),
💡 Tip: ค่า fallback ในโค้ด ('redis') จะถูกเปลี่ยนได้ผ่าน ENV ตอน deploy — ไม่ต้องแก้โค้ดถ้าจะกลับไปใช้ file ตอน local dev

2.5 เพิ่ม Trusted Proxies (สำหรับ HTTPS หลัง load balancer)

# app/Http/Middleware/TrustProxies.php
protected $proxies = '*';  // trust all (Cloud LB จัดการให้)

protected $headers =
    Request::HEADER_X_FORWARDED_FOR |
    Request::HEADER_X_FORWARDED_HOST |
    Request::HEADER_X_FORWARDED_PORT |
    Request::HEADER_X_FORWARDED_PROTO |
    Request::HEADER_X_FORWARDED_AWS_ELB;

2.6 Commit + push การแก้ไข

PS> git add cloud.yaml config/
PS> git commit -m "chore: prepare for Laravel Cloud deployment"
PS> git push origin main

3 Export ข้อมูลจาก MariaDB

⏱ ~2 นาที

เราจะ dump ทุกตารางออกเป็นไฟล์ .sql เพื่อ import เข้า MySQL บน Cloud

3.1 SSH เข้า Hostinger หรือใช้ phpMyAdmin

วิธี A: ผ่าน SSH (แนะนำ)

$ ssh u123456@your-server.hostinger.com
$ cd domains/jabphap.com/public_html

# Dump ทั้ง schema + data
$ mysqldump \
    --user=u123456_jabphap \
    --password=YOUR_DB_PASSWORD \
    --default-character-set=utf8mb4 \
    --single-transaction \
    --routines --triggers \
    --no-tablespaces \
    u123456_jabphap > jabphap-backup.sql

# เช็คขนาด
$ ls -lh jabphap-backup.sql

# Download ลงเครื่อง
PS> scp u123456@your-server.hostinger.com:domains/jabphap.com/public_html/jabphap-backup.sql .

วิธี B: ผ่าน phpMyAdmin

  1. เข้า hpanel → Databases → phpMyAdmin
  2. เลือก database ของโปรเจค (เช่น u123456_jabphap)
  3. คลิก tab Export
  4. Format: SQL
  5. Custom options: ✓ Add DROP TABLE, ✓ Disable foreign key checks
  6. Charset: utf8mb4
  7. คลิก Go → ดาวน์โหลดไฟล์ .sql
⚠️ ระวัง MariaDB-specific syntax: ก่อน import เข้า MySQL 8 ให้แก้ dump:
  • utf8mb4_uca1400_ai_ci → utf8mb4_unicode_ci (MySQL 8 ไม่มี collation นี้)
  • เอา SET @OLD_CHARACTER_SET_CLIENT ออกถ้ามี issue
  • ลบ DEFINER=... ใน triggers/views (Cloud user ไม่ตรง)

PowerShell แก้แบบ batch:

PS> (Get-Content jabphap-backup.sql) `
    -replace 'utf8mb4_uca1400_ai_ci', 'utf8mb4_unicode_ci' `
    -replace 'DEFINER=`[^`]*`@`[^`]*`', '' |
    Set-Content jabphap-backup-fixed.sql

3.2 ตรวจ dump ก่อนใช้

PS> type jabphap-backup-fixed.sql | findstr /C:"CREATE TABLE" | measure
# ควรเห็นจำนวนเท่ากับตารางในโปรเจค (~30-40 ตาราง)

6 Import Database

⏱ ~2 นาที

6.1 วิธี A: ใช้ Cloud CLI

# ติดตั้ง CLI (ครั้งเดียว)
PS> composer global require laravel/cloud-cli

# Login
PS> laravel cloud:login

# Import dump เข้า Cloud DB
PS> laravel cloud:db:import \
    --app=jabphap \
    --database=jabphap-db \
    --file=jabphap-backup-fixed.sql

6.2 วิธี B: เปิด tunnel แล้วใช้ mysql client

# เปิด tunnel ไปที่ Cloud DB
PS> laravel cloud:db:tunnel jabphap-db
# จะ print: localhost:33067 → Cloud DB

# Import (เปิด terminal ใหม่)
PS> mysql -h 127.0.0.1 -P 33067 -u cloud_user -p jabphap_db < jabphap-backup-fixed.sql

6.3 ตรวจว่า import สำเร็จ

PS> laravel cloud:db:tunnel jabphap-db
PS> mysql -h 127.0.0.1 -P 33067 -u cloud_user -p

mysql> USE jabphap_db;
mysql> SHOW TABLES;
# ควรเห็น ~30-40 ตาราง รวม bookings, photographer_availabilities, ฯลฯ

mysql> SELECT COUNT(*) FROM auth_users;
mysql> SELECT COUNT(*) FROM bookings;
⚠️ Schema กับ data: ถ้าจะให้ Laravel จัดการ schema เอง (run migrations แทน import schema จาก dump) แนะนำให้ import แค่ data tables ไม่รวม migrations table → Cloud จะ run migrations ใหม่หมดและสะอาดกว่า

🔄 Zero-Downtime Cutover จาก Hostinger

วิธีย้าย live traffic จาก Hostinger → Cloud โดยไม่ให้ user เห็น downtime

  1. เตรียมพร้อมบน Cloud (ก่อนสับ DNS)
    • Deploy + ทดสอบครบทุกข้อใน checklist ผ่าน URL jabphap-prod.laravel.cloud
    • Import ข้อมูลล่าสุดจาก Hostinger (ถ้ามี gap)
  2. Snapshot DB Hostinger ครั้งสุดท้าย (T-0) 
    $ mysqldump ... > final-snapshot.sql
  3. เปิด maintenance mode บน Hostinger (5-10 นาที) 
    $ php artisan down --message="กำลังอัพเกรดระบบ — กลับมาใน 10 นาที"
  4. Import snapshot สุดท้ายเข้า Cloud DB

    ใช้ Step 6 — แต่คราวนี้ overwrite (drop existing tables ก่อน)

  5. เปลี่ยน DNS ที่ Cloudflare
    • ลด TTL เป็น 60 วินาที ก่อนหน้านี้ 24 ชม. เพื่อให้ propagate เร็ว
    • เปลี่ยน A record เก่า → CNAME ของ Cloud
    • รอ propagate ~5 นาที
  6. ทดสอบ live 
    PS> curl -I https://jabphap.com
# Server header ต้องเปลี่ยนจาก Hostinger LiteSpeed → AWS/Cloudflare
  7. ปิด Hostinger app (ยังเก็บ server ไว้)

    เก็บ Hostinger ไว้ 7-14 วันเป็น rollback option ก่อนยกเลิก

✅ Total downtime คาดการณ์: 5-10 นาที (ตอน import snapshot สุดท้าย) — user ที่เปิดอยู่จะเห็น maintenance page สั้นๆ แล้วเข้าได้ปกติ

↩ Rollback Plan

ถ้าเกิดปัญหาหลัง cutover — ย้อนกลับ Hostinger ภายใน 5 นาที

  1. เข้า Cloudflare DNS
  2. เปลี่ยน CNAME ของ @ + www กลับเป็น A record IP เก่าของ Hostinger
  3. รอ propagate ~1-5 นาที (TTL 60s ที่เราตั้งไว้)
  4. SSH เข้า Hostinger → php artisan up เปิด site กลับ
  5. ถ้ามีข้อมูลใหม่ที่เกิดบน Cloud หลัง cutover — export กลับไป merge ที่ Hostinger
💡 Backup Plan: ตั้ง TTL DNS = 60 วินาทีล่วงหน้า 1 วัน — propagate จะเร็วทั้งขาไปและขากลับ

🔧 Troubleshooting

❌ Build fails: "composer install" memory exhausted

Cloud build container มี memory limit — แก้โดย:

  1. Commit composer.lock เข้า repo (ถ้ายังไม่มี)
  2. เพิ่ม --prefer-dist --no-progress ใน build step ของ cloud.yaml
  3. ถ้ายังไม่ได้ → upgrade compute size ชั่วคราว
❌ Migration error: "table already exists"

เพราะ import data dump พ่วง schema มาด้วย — แก้:

  1. SSH เข้า Cloud → php artisan migrate:fresh --seed (ระวัง — ลบทุกอย่าง!)
  2. หรือ import แค่ data tables (ไม่รวม migrations table)
❌ 500 Error หลัง deploy + log บอก "view not found"

view cache ค้าง — clear แล้ว rebuild:

$ laravel cloud:ssh jabphap
$ php artisan view:clear
$ php artisan config:clear
$ php artisan cache:clear
$ php artisan optimize
❌ LINE webhook signature mismatch

เกิดจาก request body ถูก middleware แก้ก่อน verify — เช็ค:

  1. App\Http\Middleware\VerifyCsrfToken ต้อง except webhooks/*
  2. Trusted Proxies = '*' (Step 2.5)
  3. LINE_CHANNEL_SECRET ใน Cloud env ตรงกับ LINE console
❌ Queue jobs ไม่ทำงาน

ตรวจ:

  1. Worker tab → Status = Running
  2. QUEUE_CONNECTION=redis ใน env
  3. SSH → php artisan queue:failed ดู failed jobs
  4. Logs tab → filter "queue" — ดู error
❌ R2/S3 upload error: "Could not resolve host"

เช็ค R2_ENDPOINT ใน Cloud env — ต้องเป็น URL เต็ม HTTPS:

R2_ENDPOINT=https://abc123.r2.cloudflarestorage.com
# ❌ ไม่ใช่ abc123.r2.cloudflarestorage.com (ไม่มี https://)
❌ Sidebar หายเหมือนเคยเจอบน Hostinger

ปัญหาเดียวกับที่เจอ — role string ใน DB ไม่ตรงกับ Model constant. แก้:

$ laravel cloud:ssh jabphap
$ php artisan tinker
>>> Admin::where('role', 'super_admin')->update(['role' => Admin::ROLE_SUPERADMIN]);
❌ Slow first request หลังไม่มี traffic

Cold start ของ Hobby tier — แก้:

  • Upgrade เป็น Production tier (warm instances)
  • หรือใช้ uptime monitor (UptimeRobot, etc.) ping ทุก 5 นาทีเพื่อกัน cold

❓ FAQ

Q: ใช้ MariaDB ต่อได้มั้ย ไม่อยากเปลี่ยน MySQL?

Cloud มีแค่ MySQL 8 และ Postgres 16 — ไม่มี MariaDB native. แต่ MySQL 8 รองรับ feature ทั้งหมดที่โปรเจคใช้ + utf8mb4 + JSON columns ดังนั้น migrate มาแล้วใช้ได้ตามปกติ ไม่ต้องแก้โค้ด

Q: เก็บภาพใน Cloud ได้มั้ย ไม่ต้องใช้ R2?

Cloud ใช้ ephemeral filesystem — ไฟล์ที่เขียน local จะหายตอน deploy ใหม่. ต้องใช้ R2/S3 เท่านั้น. โปรเจคนี้ใช้ R2 อยู่แล้วจึงไม่กระทบ ✓

Q: Pricing เพิ่มขึ้นยังไงเมื่อ traffic โต?

Cloud bill 3 ก้อน:

  • Compute: ตามขนาด instance + จำนวน hours (ถ้า auto-scale ขึ้น 2x = ค่าขึ้น 2x ชั่วคราว)
  • Database/Redis: ตามขนาด storage (fixed)
  • Bandwidth: $0.10/GB หลัง 100GB free

Set Spending Limit ใน Settings เพื่อกันบิลเกินคาด

Q: ย้ายออกจาก Cloud ไปที่อื่นได้มั้ย?

ได้สบาย — code ยังเป็น Laravel มาตรฐาน. Steps:

  1. Export DB ผ่าน cloud:db:export
  2. git clone repo
  3. Setup ที่ Forge/DO/AWS ของใหม่
  4. Import data + เซ็ต ENV

มีแค่ cloud.yaml ที่จะไม่จำเป็นในที่อื่น (แต่ปล่อยไว้ก็ไม่เสียหาย)

Q: รองรับ multi-environment (staging + production) มั้ย?

ได้ — สร้าง app แยกได้ 2 ตัว แต่ละตัวต่อกันคนละ branch:

  • jabphap-prod ↔ main branch
  • jabphap-staging ↔ staging branch (free Sandbox tier ก็ได้)
Q: SSH เข้าได้ไหม? ต้องเปลี่ยน workflow มั้ย?

SSH ได้ผ่าน CLI: laravel cloud:ssh jabphap — เข้าไปได้ทุก artisan command, tinker, ดูไฟล์ ฯลฯ. แต่ ห้ามแก้ code ผ่าน SSH เพราะ deploy ใหม่จะทับ — ทุกการแก้ต้องผ่าน git commit + push

Q: ใช้ /admin/deployment ที่สร้างไว้ก่อนหน้านี้ได้มั้ย?

ไม่จำเป็นแล้ว — Cloud จัดการ schema/migration ผ่าน cloud.yaml ให้อัตโนมัติ. แต่หน้า admin/deployment ก็ยังใช้ดู status ของ ENV ได้
